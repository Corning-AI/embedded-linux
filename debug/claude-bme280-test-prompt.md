**【背景信息】**
我们在 NXP i.MX8MP EVK 上调试 GY-BME280 (I2C 0x76/0x77) 时遇到双盲问题：`i2cdetect` 均返回 NACK。目前怀疑点在于：
1. 扩展口 J21/J22 上的 `VEXT_3V3` 根本没有实际输出 3.3V 电压（尽管 GPIO 使能已置高）。
2. J21 上的 TXS0102 双向电平转换器与 GY-BME280 模块上自带的分立 MOS 电平转换电路发生了严重的阻抗不匹配或锁死。
3. 总线发生了电气隔离或假死。

### 阶段 1：内核总线死锁与中断观测
请在终端中执行以下命令来观察 I2C 控制器在探测时的硬件中断反映：
1. 记录探测前的 i2c 中断：`cat /proc/interrupts | grep i2c` 
2. 对 J21 (I2C3) 执行探测：`i2cdetect -y 2`
3. 再次记录中断：`cat /proc/interrupts | grep i2c`
4. 检查内核日志：`dmesg | tail -n 20`
**分析要求**：对比两次中断数。如果 `30a40000.i2c` 的中断数在探测期间**一点都没涨**，且 dmesg 出现 NXP I2C Recovery 报错，这说明物理总线被严重拉低或锁死。如果中断数正常激增但探测全 NACK，说明物理波形发出了但传感器没响应或被电平芯片隔离。请告诉我你的结论。

### 阶段 2：引脚复用夺权排查
以防万一 `gpioset` 虽然设置了引脚，但设备树的底层 pinmux 并没有正确把 I2C 管脚分配给设备，或者被其他驱动霸占。
请执行：
`cat /sys/kernel/debug/pinctrl/30330000.pinctrl-snvs/pinmux-pins | grep i2c`
以及 
`cat /sys/kernel/debug/pinctrl/30330000.pinctrl-snvs/pinconf-pins | grep i2c`
**分析要求**：确认 I2C3 和 I2C5 的相关 pad（引脚焊盘）当前的 `owner` 是否确切是 `30a40000.i2c` 和 `30ad0000.i2c` 控制器，有没有出现 `UNCLAIMED` 或被无关驱动意外占用的情况？
